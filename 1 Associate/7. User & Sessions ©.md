## [Web Applications](https://docs.dynatrace.com/docs/observe/digital-experience/web-applications)

Is a front-end of an application that can be accessed via a browser or mobile app. Tracks the user's experience when they interact with a website or an app.

* **User Actions** Clicks or user interactions.
* **XHR Actions** Background requests that update parts of a page (like clicking "Next" in a photo gallery).
* **Custom Actions** Business-specific actions that are being tracked manually (like PDF downloads, using the Dynatrace JavaScript API (OpenKit)).

By default, Dynatrace groups all web traffic under **"My Web Application"**.

## [Digital Experience Monitoring (DEM)](https://docs.dynatrace.com/docs/observe/digital-experience)

Provides monitoring of user interactions across web, mobile, and other digital touchpoints, ensuring applications are available, functional, and performing well.

Key components of DEM are **[Real User Monitoring (RUM)](https://claude.ai/chat/1dfa9358-b60d-4f66-91d7-cb12546a98d3#%5BReal%20User%20Monitoring%20\(RUM\)%5D\(https%20//docs.dynatrace.com/docs/observe/digital-experience/web-applications/initial-setup/define-your-applications-via-the-my-web-application-placeholder\))** and **[Synthetic Monitoring](https://claude.ai/chat/1dfa9358-b60d-4f66-91d7-cb12546a98d3#%5BSynthetic%20Monitoring%5D\(https%20//docs.dynatrace.com/docs/license/capabilities/digital-experience-monitoring/browser-monitor-clickpath\))**.

## [Real User Monitoring (RUM)](https://docs.dynatrace.com/docs/observe/digital-experience/web-applications/initial-setup/define-your-applications-via-the-my-web-application-placeholder)

Tracks and analyses real-time user interactions within applications, measuring the performance of a front-end application from the perspective of an actual user. Focuses on **performance**, **reliability**, and **responsiveness**.

Can monitor **websites**, **mobile apps**, and **custom applications** (like smart devices).

When OneAgent monitors the web server, it instructs it to insert a **JavaScript Snippet** into the page response. This snippet collects monitoring data about the user's experience — a process called **"Auto Injection"**.

Dynatrace routes traffic to the correct application using **detection rules**, typically based on URL patterns. Admins set these rules to ensure each application's data is placed in the right location.

Dynatrace can capture **IP address** (maskable for privacy), **browser type** (Chrome, Edge, Safari, etc.), and **user type** (real user, synthetics, bots).

#### Apdex — Performance User Experience Score

A widely used scoring system for measuring user experience.

* **0 to 3 seconds** → user is considered **satisfied**
* **3 to 15 seconds** → experience is **tolerable**
* **over 15 seconds** (or if the action results in an error like a 404) → considered **frustrating**

> Apdex of **1** is the ultimate goal — the best possible user experience.

#### Key User Actions

You can mark specific user actions as **Key User Actions** to:

* Customise Apdex thresholds for important pages (e.g., require login pages to load under 2 seconds)
* Add them to dashboards for easy monitoring
* Enable long-term storage of that user action's data

#### Waterfall Analysis

Shows the loading sequence of page elements (scripts, images, CSS). Helps pinpoint exactly what's slowing down a user's action. Based on W3C timing standards — no proprietary metrics.

#### Error Analysis

Breaks down JavaScript or request-related errors tied to specific user actions. Useful for identifying bugs affecting user experience.

## [Mobile RUM](https://docs.dynatrace.com/docs/observe/digital-experience/mobile-applications)

Extends Real User Monitoring to native mobile applications. Uses an SDK integrated directly into the mobile app to capture user interactions, crashes, and performance data.

## [Agentless RUM](https://docs.dynatrace.com/docs/observe/digital-experience/web-applications/initial-setup/set-up-agentless-real-user-monitoring)

An alternative method used when OneAgent cannot be installed on a web server — for example, when the app's technology doesn't support auto-injection, or security policies prevent it.

With this method, the **Dynatrace JavaScript snippet must be manually inserted** into each relevant page (login, homepage, search, etc.). If the tag isn't on a specific page, it will not be monitored.

* **Load Actions** — triggered when a full page is loaded (home page, login page, search results).
* **XHR Actions** — triggered when part of the page updates without a full reload (filters, product images).
* **Custom Actions** — not captured automatically; defined using the Dynatrace JavaScript API.

For non-web, non-mobile applications, **Dynatrace OpenKit** allows monitoring of thick clients using Java, .NET, or C code.

> [!NOTE]
> Setting up custom actions is typically handled by the **development team**, in coordination with the **application team** who decides what actions to track.

## User Sessions

#### [Session Segmentation](https://docs.dynatrace.com/docs/observe/digital-experience/session-segmentation)

Sessions can be segmented and filtered to focus on specific groups of users or behaviour patterns, enabling more targeted analysis.

#### [User Behaviour Analysis](https://docs.dynatrace.com/docs/observe/digital-experience/web-applications/analyze-and-use/user-behavior-analysis)

Captures **page loads**, **payments**, **errors**, **page navigation**, and more. Displays key performance metrics regarding user behaviour, such as bounces, top landing and exit pages, conversions, and many more.

Dynatrace can **tag user sessions** with identifiers such as usernames or email addresses. For example, if a web page says "Welcome Ryan", Dynatrace can be configured to recognise "Ryan" as the user.

> [!NOTE]
> Tagging setup is typically handled by someone familiar with the application's code.

Along with user actions, Dynatrace automatically captures **technical and environment data**, including: tagged identifiers (usernames, emails), applications visited, screen resolution, browser and operating system, device type, and more.

#### Conversion Goals

Dynatrace lets you define **conversion goals** — specific events that indicate a successful session. Common types include:

* Visiting a specific page (e.g. "Payment Finish")
* Performing a certain number of actions
* Staying on the site for a minimum amount of time

> It can take about **30 minutes** for a new conversion goal to show results in Dynatrace. For a session to count toward a conversion goal, the **session must be marked as completed**.

#### When Is a User Session Considered Completed?

1. After **35 minutes of browser inactivity**
2. When the **browser window is closed** (closing a tab does not end the session)
3. After **8 hours of activity**
4. After **more than 200 user actions** (a new session starts on the 201st)
5. When the **JavaScript API** calls `dtrum.endSession()` (e.g. on user logout)

#### Licensing and Session Counts

Each user session typically consumes **one license** in Dynatrace.

**Bounce sessions** (a single action, such as loading a page without any interaction) may **not be counted** toward license consumption.

> [!TIP]
> Specific licensing rules and thresholds may differ depending on your plan. Consult your account representative for guidance.

#### Privacy and Data Masking

Dynatrace RUM can mask sensitive user data, including **email addresses**, **IP addresses**, and **physical locations**. If you see the word **"masked"** in a session, it means the data is either hidden by policy or you don't have the proper user permissions.

## [Synthetic Monitoring](https://docs.dynatrace.com/docs/license/capabilities/digital-experience-monitoring/browser-monitor-clickpath)

Simulates user interactions with applications using automated scripts, identifying performance issues **before they affect real users**. Monitors web applications **24/7**, even when no actual users are online.

> [!NOTE]
> You do **not** need OneAgent to use Synthetic Monitoring — everything runs from public Dynatrace locations.

There are **3 types** of synthetic monitors:

* **Browser Monitors** Simulate loading a single page (e.g. login or home page). Ideal for basic uptime and performance checks.
* **Browser Clickpath Monitors** Simulate a full user journey (e.g. log in → search → purchase). Useful for testing entire workflows across multiple pages.
    
    > Requires the **Dynatrace Synthetic Recording** browser extension for Chrome.
    
* **HTTP Monitors** Check API endpoints or URLs that don't load a visual page. Like sending a "ping" or "curl" request to verify availability.

Synthetics can be configured to notify about: **downtime**, **slow page performance**, and **failed logins or broken user flows**.

> After setup, it can take up to **15–30 minutes** for a synthetic monitor to become active, especially for complex clickpaths.

#### Customisation Options

* **Frequency and Locations**
	* Change how often the synthetic runs and from which region.
* **Outage Handling**
	* Define how many consecutive failures should trigger a problem (e.g. 3 failures in a row).
* **Performance Thresholds**
	* Alert when a page exceeds a defined load time (e.g. over 3 seconds).
* **On-Demand Execution**
	* Manually trigger a synthetic at any time, or via the Dynatrace API.

#### Global Coverage

Synthetics run from **public Dynatrace locations** around the world. You can choose specific test locations (U.S., Europe, Australia, etc.) to see how your app performs from different regions.

#### Private Synthetic Locations

For applications in private networks not accessible from public Dynatrace locations:

* **Option 1: Whitelist Dynatrace IPs**
	* Allow specific Dynatrace Synthetic IPs through your network.
* **Option 2: Private Synthetic Locations**
	* Set up synthetics inside your network using **ActiveGate**.

#### Credential Vault

Usernames and passwords needed for synthetic logins can be stored securely in the **Credential Vault**, keeping login details private and allowing others to build synthetics without seeing sensitive information.

## Session Replay

Allows you to visually **reconstruct a user's journey** through your application — without recording any actual video or screen content. It's a reconstruction based on collected data, not a live recording.

Particularly helpful for identifying tricky front-end issues, such as a button that appears to do nothing, or endless spinners caused by a failing element.

By default, Session Replay **masks sensitive user data** — text, form fields, images, and dropdown values. The page layout is visible, but actual content is hidden unless explicitly unmasked.

> **Passwords and other private data will ALWAYS remain masked** by design, regardless of configuration.

Session Replay is useful for:

* Troubleshooting hard-to-reproduce frontend issues
* Understanding exactly what a user did before an error occurred
* Improving support and development response times
* Maintaining full visibility while protecting privacy
