## Custom Services ©
For application services **not built on standard technologies** — they may not be recognised out of the box.

> [!NOTE]
> #### When do you need custom services?
> Two common use cases:
> 1. Services built on **unsupported technologies** that Dynatrace doesn't detect automatically.
> 2. **Microservices** used by a larger application that aren't exposed via standard communication technologies.

A **custom service** is defined by manually specifying a **method**, **class**, or **interface** as the service's entry point.

## Request Attributes ©
Dynatrace automatically tracks all requests end-to-end and monitors the services underlying each transaction.

* Predefined attributes are captured automatically
* **Custom request attributes** can be configured to improve **filtering and analysis** of web requests

> [!TIP]
> Use custom request attributes to capture values like a customer tier, order type, or region — anything useful for filtering that isn't captured by default.

## Deep Monitoring
OneAgent supports **deep monitoring** for many technologies and frameworks.

* Provides further visibility into services
* For example: additional code-level detail in traces if OneAgent features are enabled

Configure at: **Settings → Server-side service monitoring → Deep monitoring**

## Process Group Detection ©
Dynatrace automatically merges related processes into **process groups** — logical clusters of processes that belong to the same application or deployment unit and perform the same function across multiple hosts.

Detection is controlled by a **default set of detection rules**.

* **Simple rules** — based on basic process properties
* **Advanced rules** — more granular matching conditions

> [!TIP]
> #### Why host groups affect process groups
> Processes running the same software across different host groups are **separated into different process groups** by Dynatrace. This prevents production and non-production data from mixing.
> More → **[1. Platform Health & Configuration](1__Platform_Health_and_Configuration.md)**

## Declarative Process Grouping ©
In large, dynamic environments, the number of processes can become overwhelming. Dynatrace automatically monitors:

* Process groups of a **known technology type**
* Process groups that **consume significant resources**

For everything else — use **declarative process grouping**.

> Declarative process grouping enables **fine-grained rules** to match any specific process group, including:
> Processes of an **unknown technology type**
> Processes that **don't consume significant resources**

## Process Availability Monitoring ©
Monitor the availability of **key processes** on your hosts.

1. Define a **monitoring rule** for a process
2. When a matching process is **missing on a host**, Dynatrace issues an **alerting event**

Configuration options:
* **Rule scope**
	* Which hosts or host groups the rule applies to
* **Rule definition**
	* Conditions to match the target process
* **Management**
	* Update or remove rules as your environment evolves

## Extension Governance
Your organisation may create and manage extensions to centralise and automate data collection configurations. Monitoring extension health is key to ensuring data collection functions as expected.

#### Three Ways to Manage Extensions ©
1. **Dynatrace Extensions app** (latest Dynatrace)
2. **Dynatrace Hub**
3. **Dynatrace API**

> [!TIP]
> Extensions can also be browsed in the **Dynatrace Hub** — the "app store" for Dynatrace. Most extensions come with a pre-made dashboard.

## ActiveGate Groups ©
Used for **bulk management** of ActiveGates, particularly for managing **Extensions 2.0**.

> [!IMPORTANT]
> An ActiveGate running a **remote extension** must belong to an **ActiveGate group**.
> Dynatrace uses the group to instruct the extension **where it should run**.

If you plan to use a **single ActiveGate** for a remote extension: assign it to a **dedicated group** containing only that ActiveGate.
