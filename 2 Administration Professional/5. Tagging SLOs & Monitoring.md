## Tagging vs. Metadata ©
Two closely related but fundamentally different concepts.

| | Tags | Metadata |
|---|---|---|
| **What they are** | Labels or markers | Key/value pairs |
| **Relationship to entity** | Do NOT belong to an entity (external) | Inherent to any monitored entity |
| **Primary use** | Organising entities | Defining extra information for entities |
| **Created by** | Manually, auto-rules, API | Dynatrace auto-detects; can define custom |
| **Can create tags from?** | — | Yes — tags can be based on metadata |

> [!NOTE]
> Dynatrace automatically detects and displays metadata values related to processes. These can then be used to **automate tagging**.

#### Tag Types
* **Manually applied** — added by a user
* **Automatically applied** — via auto-tagging rules
* **API-created** — pushed programmatically

## Auto-Tagging & Properties
Dynatrace supports automated tagging in Kubernetes environments using:
* **Kubernetes labels**
* **Kubernetes annotations**
* **Environment variables**

Custom metadata can also be defined for your organisation's unique needs.

> [!TIP]
> Using auto-tagging rules keeps large, dynamic environments organised without manual effort. Combine with management zone rules for maximum scalability.

## Naming Rules
Customise how entities appear in Dynatrace for improved clarity (e.g., `svc-12345` → `OnlineBoutique-LoginService`).

Built-in rules define default names — **custom rules enhance or override** them.

Naming rules can be set for:
* **Services** — built-in rules based on application deployment properties; custom rules available
* **Process groups** — auto-detected names may be too generic; define naming rules to match your standards
* **Hosts** — set custom host names in dynamic environments
* **Requests** — set up request naming rules

#### Process Group Naming Rules
Steps to define or enrich naming rules:
1. Navigate to **Settings → Processes and containers → Process group naming**
2. Define rules with conditions matching your process properties

## Service-Level Objectives (SLOs) ©
SLOs combine multiple metrics and indicators to closely monitor the reliable delivery of a service against an operational goal.

#### Components of an SLO ©
| Component | Description |
|---|---|
| **Service-Level Indicator (SLI)** | The metric used to measure successful service delivery |
| **Target** | The planned goal to achieve |
| **Evaluation period** | Standardises communication about SLO results |

#### Configuring SLOs in Dynatrace
Two ways to create an SLO:
1. **SLO Wizard** — select from preconfigured Dynatrace templates for common use cases
2. **Custom SLO definition** — define your own SLI, target, and evaluation period

When selecting entities for an SLO, you can filter by:
* **Tag**
* **Entity type**

## Real-User Monitoring (RUM)
RUM provides **performance analysis in real time** for all user actions and their impact.

* Tracks user behaviour including all user actions taken
* Provides insight into performance impact per action
* Powers **Session Replay** for visual playback of user sessions

> More on RUM data privacy → **[2. Data Privacy & Security](2__Data_Privacy_and_Security.md#RUM%20Data%20Privacy%20Settings)**

## Application Detection Rules ©
Application detection rules define **which transactions compose your RUM applications**.

* User actions detected by Dynatrace are **grouped together** to represent your application from the user perspective
* Enables **standalone monitoring and analysis**

Best practices:
* Verify detection rules detect traffic from the **correct URLs**
* Ensure rules accurately reflect the applications they're designed to detect

## Synthetic Monitoring
Completes the picture of customer experience by monitoring availability and performance **around the world and around the clock**.

#### Monitor Types ©
| Type | Description |
|---|---|
| **Single-URL browser monitor** | Simulated user visiting app via a modern browser |
| **Browser clickpath** | Multi-step user interaction simulation |
| **HTTP monitor** | API / endpoint availability check |
| **NAM monitor** | ICMP, TCP, DNS protocols only |

#### Frequency Options ©
* Every **5, 10, 15, or 30 minutes**
* Every **1, 2, or 4 hours**
* **On demand** only

#### Execution Calculation ©
```
Executions per hour = frequency multiplier × number of locations
```

> Example: **3 locations** × every **15 min** (= 4×/hour) = **12 executions/hour**

Executions are **evenly spaced** within the interval.

#### Limits ©
* Maximum **5,000 NAM monitors** per environment
* Maximum **5,000 on-demand executions** per minute per environment
* Monitor name limited to **500 characters**
* Up to **64 key-value metadata pairs** per on-demand batch

#### Private Synthetic Locations ©
Require an **Environment ActiveGate** and access to tested resources.

Recommended specs:
* Minimum **8 GB RAM** (4 GB may cause memory shortages)
* Minimum **25 GB free disk space**
* Chromium auto-update can be enabled/disabled per location

## Synthetic Outage Handling ©
Outage handling settings determine what happens during monitor failures.

* Default outage handling can be configured at the **environment level** for all browser or HTTP monitors
* Can **disable problem generation** during known outages

## Synthetic Performance Thresholds ©
Enable performance thresholds to track SLA compliance.

Set a threshold **in seconds** for:
* The **monitor as a whole**
* **Each individual action**

Useful for enforcing SLA performance time requirements.
